## 🔴 Critical Issues (Phải sửa ngay)

### 1. Hardcode mật khẩu SQL Server trong source code
- **File:** `src/main/config/app-config.ts:12-13`
- **Triệu chứng:** Mật khẩu SQL Server `sa / Pnj@12345` được viết trực tiếp trong code.
- **Nguy hiểm:** Nếu code bị lộ ra ngoài (GitHub, máy bị hack...), kẻ xấu có quyền truy cập **TOÀN BỘ** database SQL Server với quyền cao nhất (`sa` = System Administrator).
- **Thêm nữa:** Mật khẩu device ZKTeco (`938948`) cũng bị hardcode tại dòng 23.
- **Cách sửa:** Chuyển sang đọc từ file `.env` và KHÔNG commit `.env` lên Git. Thêm `.env` vào `.gitignore`.

```diff
# app-config.ts
- password: process.env.WISEEYE_SQL_PASSWORD ?? 'Pnj@12345',
+ password: process.env.WISEEYE_SQL_PASSWORD ?? '',
```

### 2. Hardcode session encryption key
- **File:** `src/main/config/app-config.ts:10`
- **Triệu chứng:** `sessionEncryptionKey: 'ccpro-desktop-session-key'` — khóa mã hóa session giống nhau trên MỌI máy.
- **Nguy hiểm:** Nếu biết key này, kẻ xấu có thể **giải mã session** của tất cả nhân viên → giả mạo đăng nhập.
- **Cách sửa:** Sinh khóa ngẫu nhiên cho mỗi máy khi cài đặt lần đầu, lưu vào `electron-store` hoặc `.env`.

---

### 3. SQL Injection trong Database Init
- **File:** `src/main/db/init.ts:10`
- **Triệu chứng:** `IF DB_ID(N'${databaseName}') IS NULL` — tên database được chèn trực tiếp vào câu SQL bằng string interpolation.
- **Nguy hiểm:** Nếu giá trị env `CCPRO_APP_DATABASE` bị chỉnh sửa thành code SQL độc, hacker có thể **xóa sạch hoặc kiểm soát database**.
- **Cách sửa:** Validate tên database chỉ chứa ký tự hợp lệ (alphanumeric + underscore).

```typescript
const safeName = /^[a-zA-Z0-9_]+$/.test(databaseName)
if (!safeName) throw new Error('Invalid database name')
```

---

### 4. Unrestricted `shell.openExternal(url)` — mở mọi URL
- **File:** `src/main/ipc/register-handlers.ts:337`
- **Triệu chứng:** IPC handler `app:open-external` mở BẤT KỲ URL nào từ renderer mà không kiểm tra.
- **Nguy hiểm:** Nếu renderer bị XSS, kẻ xấu có thể chạy `file:///C:/...` hoặc `smb://...` để truy cập file cục bộ hoặc chạy phần mềm độc.
- **Cách sửa:** Chỉ cho phép mở URL `https://`.

```typescript
ipcMain.handle('app:open-external', async (_event, url: string) => {
  if (!url.startsWith('https://')) return
  const { shell } = await import('electron')
  shell.openExternal(url)
})
```

## 🟡 Warnings (Nên sửa)

### 5. Không có Rate Limiting cho đăng nhập
- **File:** `src/main/services/auth-service.ts`, `admin-auth-service.ts`
- **Triệu chứng:** Không giới hạn số lần đăng nhập sai. Có thể thử mật khẩu liên tục.
- **Hậu quả:** Kẻ xấu dùng chương trình tự động thử **hàng nghìn mật khẩu** cho đến khi đoán đúng (brute-force attack).
- **Cách sửa:** Thêm bộ đếm và khóa tạm sau 5 lần sai:
  - Sai 5 lần → khóa 5 phút
  - Sai 10 lần → khóa 30 phút

---

### 6. Mật khẩu mặc định bị LOG ra console
- **File:** `src/main/db/init.ts:478, 503`
- **Triệu chứng:** `console.log('[init] Created default admin account: admin / ccpro@2026')` — mật khẩu admin mặc định hiện rõ trong log.
- **Hậu quả:** Ai có quyền đọc log đều biết mật khẩu admin mặc định.
- **Cách sửa:** Xóa password khỏi log message.

---

### 7. `sandbox: false` trong BrowserWindow
- **File:** `src/main/index.ts:59`
- **Triệu chứng:** Renderer process chạy với quyền cao hơn cần thiết.
- **Hậu quả:** Nếu renderer process bị tấn công (XSS), hacker có thể truy cập tài nguyên hệ thống dễ dàng hơn.
- **Cách sửa:** Nghiên cứu bật `sandbox: true`, nhưng cần kiểm tra tương thích với preload script.

---

### 8. Mật khẩu mặc định user = Mã nhân viên
- **File:** `src/main/services/auth-service.ts:91`
- **Triệu chứng:** `password === employeeCode` — khi chưa đổi mật khẩu, mật khẩu chính là mã nhân viên.
- **Hậu quả:** Ai biết mã nhân viên (thường công khai) có thể đăng nhập vào tài khoản người khác.
- **Cách sửa:** Đây là acceptable nếu có flow bắt buộc đổi mật khẩu lần đầu (đã có `isFirstLogin`). Tuy nhiên, nên thêm cảnh báo rõ ràng và theo dõi những user chưa đổi password.

---

### 9. `dangerouslySetInnerHTML` trong frontend
- **File:** `src/renderer/src/pages/admin-users-page.tsx:521`
- **Triệu chứng:** Dùng `dangerouslySetInnerHTML` để chèn CSS inline.
- **Hậu quả:** Rủi ro thấp vì nội dung là static CSS hardcode, nhưng đây là bad practice. Nếu sau này có ai sửa để chèn biến động, sẽ tạo lỗ hổng XSS.
- **Cách sửa:** Chuyển CSS sang file `styles.css` hoặc dùng CSS modules.
